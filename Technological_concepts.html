Technological Concepts

Webhooks

A webhook is a mechanism that allows one application to automatically send data to another application when a specific event happens. Instead of asking a server again and again whether something changed, the receiving system gives the sending system a URL, and the sending system calls that URL when the event occurs.

In simple terms, polling means "keep checking for updates," while a webhook means "notify me immediately when an update happens." This makes webhooks useful for event-driven communication between systems.

Why webhooks are used

Webhooks are used when one system needs to react quickly to changes in another system. They reduce unnecessary network requests and help keep systems synchronized in near real time.

Common examples:

  • GitHub sends a webhook when code is pushed to a repository.
  • A payment gateway sends a webhook when a payment succeeds or fails.
  • An e-commerce platform sends a webhook when an order is created.
  • A messaging platform sends a webhook when a new message arrives.

How webhooks work

  1. An application exposes an HTTP endpoint such as /webhooks/github.
  2. The endpoint URL is registered in another system.
  3. An event happens in the source system.
  4. The source system sends an HTTP request, usually POST, to the registered URL.
  5. The destination system validates the request and processes the payload.
  6. The destination system returns an HTTP response code to acknowledge receipt.

The payload often contains structured JSON data describing the event, such as event type, timestamp, object identifiers, and related metadata.

Main components of a webhook

  • Event source: The system that generates the event.
  • Webhook URL: The endpoint that receives the notification.
  • Event payload: The data sent with the webhook request.
  • Headers: Metadata such as content type, signature, or event name.
  • Receiver logic: The code that validates and handles the incoming request.

Webhook example flow

Assume GitHub is connected to a deployment service. When a developer pushes code to the main branch, GitHub triggers a webhook and sends repository and commit details to the deployment service. The deployment service verifies the request, checks the branch, and starts the deployment pipeline automatically.

Webhook payload example

{
  "event": "push",
  "repository": "sample-project",
  "branch": "main",
  "pushed_by": "developer1",
  "timestamp": "2026-04-05T10:30:00Z"
}

Real payloads are usually larger and include IDs, URLs, sender information, and detailed object data.

Webhook vs API polling

  • Polling: The client repeatedly asks the server whether new data is available.
  • Webhook: The server pushes data to the client when an event occurs.

Comparison:

  • Polling can waste requests if nothing changes.
  • Webhooks are more efficient for event-based updates.
  • Polling is simpler when the receiver cannot expose a public endpoint.
  • Webhooks are better when timely updates are important.

Advantages of webhooks

  • Near real-time event delivery.
  • Less unnecessary traffic compared to frequent polling.
  • Good fit for automation workflows.
  • Useful for integrating independent services.

Challenges and risks

  • The receiving endpoint must be available and reachable.
  • Requests can be retried, so duplicate processing must be handled safely.
  • Payloads must be validated because the endpoint is exposed over HTTP.
  • Failures in downstream systems can delay or block processing.
  • Order of events is not always guaranteed.

Security best practices

  • Use HTTPS so payloads are encrypted in transit.
  • Verify signatures or shared secrets sent in request headers.
  • Validate the payload schema before processing it.
  • Authenticate the event source whenever the provider supports it.
  • Log requests and failures for debugging and auditing.
  • Protect against replay attacks when timestamps or delivery IDs are available.

Reliability best practices

  • Return a fast success response after basic validation.
  • Process heavy work asynchronously through queues or background jobs.
  • Store delivery IDs to detect duplicate events.
  • Implement idempotent handlers so repeated deliveries do not break data.
  • Keep retry handling in mind because many providers retry failed deliveries.

Typical HTTP response codes

  • 200 OK: Event received and accepted.
  • 202 Accepted: Event received and queued for processing.
  • 400 Bad Request: Invalid payload or missing required data.
  • 401 Unauthorized or 403 Forbidden: Authentication or signature validation failed.
  • 500 Internal Server Error: Receiver failed to process the event.

Where webhooks are commonly used

  • CI/CD pipelines
  • Payment status updates
  • CRM and marketing automation
  • Chat and notification systems
  • Inventory and order management
  • Third-party SaaS integrations

Summary

A webhook is an event notification mechanism in which one system sends an HTTP request to another system when something important happens. It is widely used for automation, integrations, and real-time updates. To use webhooks effectively, focus on security, validation, retry handling, and idempotent processing.
