This section groups common Git concepts you use while working with local changes, branches, and commit history.

git stash temporarily saves your uncommitted changes so you can switch branches or pull updates without committing unfinished work.

When stash is useful

  • You need to change branches quickly.
  • You want to pull the latest code without mixing local edits.
  • You are experimenting and do not want to commit partial work.

Common stash commands

git stash
git stash push -m "work in progress"
git stash list
git stash apply
git stash pop
git stash drop

How it works

Git stores the current tracked changes in a hidden stack-like list. You can reapply them later with apply or restore and remove them using pop.

Example flow

git stash push -m "login form changes"
git switch main
git pull
git switch feature/login
git stash pop

Git identity is the name and email attached to your commits. This is different from authentication. Identity controls how your commits appear in Git history and on GitHub.

1. Set identity for all repositories on your machine

Use global config when you want the same identity everywhere by default. 

git config --global user.name "Your Name"
git config --global user.email "you@example.com"

2. Set identity only for the current repository

Use local config when one repository should use a different name or email. 

git config user.name "Project Specific Name"
git config user.email "project@example.com"

3. Check which identity is active

git config user.name
git config user.email
git config --list --show-origin

--show-origin helps you see whether the value comes from the system, global, or local configuration file.

4. Common ways to manage identity

Use this when you want one identity for every repository on your machine.

Setup:

git config --global user.name "Your Name"
git config --global user.email "you@example.com"

Manage or update it later:

git config --global user.name "Updated Name"
git config --global user.email "updated@example.com"

Check the current global identity:

git config --global user.name
git config --global user.email

Use this when most repositories use one identity, but some projects need a different one.

Setup the default identity first:

git config --global user.name "Your Default Name"
git config --global user.email "default@example.com"

Then override it inside a specific repository:

git config user.name "Work Name"
git config user.email "work@example.com"

Manage the repository-specific identity:

git config user.name "New Repo Name"
git config user.email "new-repo@example.com"

Remove the local override and go back to the global identity:

git config --unset user.name
git config --unset user.email

Check whether the repository is using local or global values:

git config --list --show-origin

Use this when repositories are grouped into folders such as work and personal, and you want Git to pick the correct identity automatically.

Add conditional rules in your global config:

git config --global includeIf.gitdir:~/work/.path ~/.gitconfig-work
git config --global includeIf.gitdir:~/personal/.path ~/.gitconfig-personal

Create the identity file for work:

[user]
  name = Your Work Name
  email = work@example.com

Create the identity file for personal repositories:

[user]
  name = Your Personal Name
  email = personal@example.com

Manage these identities by editing the matching config file later.

Check that Git is loading the correct file:

git config --list --show-origin

If a repository is in the wrong folder, move it to the correct folder or add a local override inside that repository.

5. Good practice

Set a global default first, then override it locally for repositories that need a different identity. If you regularly switch between work and personal repositories, conditional config is cleaner.

Use separate SSH keys and a dedicated SSH config so Git can decide which GitHub account to use for each repository.

1. Create 2 SSH keys

ssh-keygen -t ed25519 -C "personal@example.com"
ssh-keygen -t ed25519 -C "work@example.com"

Save them with different names such as id_ed25519_personal and id_ed25519_work.

2. Add the keys to the SSH agent

ssh-add ~/.ssh/id_ed25519_personal
ssh-add ~/.ssh/id_ed25519_work

3. Add both public keys to GitHub

Copy each .pub file and add it to the correct GitHub account from Settings > SSH and GPG keys.

4. Configure SSH aliases

Create or update ~/.ssh/config:

Host github-personal
  HostName github.com
  User git
  IdentityFile ~/.ssh/id_ed25519_personal

Host github-work
  HostName github.com
  User git
  IdentityFile ~/.ssh/id_ed25519_work

5. Clone repositories with the correct alias

git clone git@github-personal:your-personal-username/repo-name.git
git clone git@github-work:your-work-username/repo-name.git

6. Set the right Git identity per repository

git config user.name "Your Personal Name"
git config user.email "personal@example.com"
git config user.name "Your Work Name"
git config user.email "work@example.com"

7. Test both accounts

ssh -T git@github-personal
ssh -T git@github-work

Each command should confirm authentication for the matching GitHub account.
