Software Testing

Testing validates that software behaves as expected, prevents regressions, and improves confidence during delivery.

The sections below use nested accordions so you can expand from broad testing concepts into techniques, execution types, and practical best practices.

Testing can be grouped by technical scope, business purpose, and execution style.

  • Unit Testing: Verifies small isolated units such as functions, classes, or methods.
  • Integration Testing: Validates interaction between modules, services, databases, or APIs.
  • System Testing: Checks the complete application as an integrated whole.
  • End-to-End (E2E) Testing: Validates complete user journeys across the UI, backend services, and data layer in production-like flows.
  • Acceptance Testing: Confirms the product meets business needs and release criteria.

  • Functional Testing: Confirms features behave according to requirements.
  • Regression Testing: Ensures existing features still work after code changes.
  • Smoke Testing: Performs a quick high-level validation on major flows.
  • Sanity Testing: Narrowly checks a specific fix or updated area.
  • Exploratory Testing: Uses human investigation to uncover unexpected issues.

  • Performance Testing: Measures response time, throughput, and system stability.
  • Security Testing: Finds vulnerabilities, access issues, and data exposure risks.
  • Usability Testing: Evaluates how easily users can complete tasks.
  • Compatibility Testing: Validates behavior across devices, browsers, and environments.

  • Manual Testing: Best for exploratory, usability, and rapidly changing features.
  • Automated Testing: Best for repeatable validations, regression coverage, and CI pipelines.
  • End-to-End Automation: Useful for validating critical business flows such as login, checkout, or form submission across integrated systems.
  • Balanced Strategy: Automate stable and repetitive checks, keep human effort for discovery and UX judgment.

  • Equivalence Partitioning: Divide inputs into meaningful groups and test one value from each.
  • Boundary Value Analysis: Focus on edges where defects often occur.
  • Decision Table Testing: Validate combinations of rules and outcomes.
  • State Transition Testing: Verify behavior when moving between states.

  1. Understand requirements and identify risks.
  2. Plan the test scope, environments, and strategy.
  3. Design test cases and prepare test data.
  4. Execute tests and document actual results.
  5. Log defects, retest fixes, and run regression checks.
  6. Summarize coverage, quality status, and release readiness.

  • Write clear, independent, and repeatable test cases.
  • Prioritize high-risk and business-critical flows first.
  • Keep test data realistic and environment setup stable.
  • Integrate automated tests into CI/CD pipelines.
  • Review flaky tests quickly to preserve trust in automation.
  • Track both defect counts and escaped production issues.

  • Unit: JUnit, NUnit, Jest, xUnit
  • UI Automation: Selenium, Playwright, Cypress
  • API Testing: Postman, Rest Assured, Newman
  • Performance: JMeter, k6, Gatling
  • Defect Tracking: Jira, Azure DevOps

A/B testing compares two or more versions of a product experience with real users to learn which version performs better against a measurable goal.

  • Control: The existing version used as the baseline.
  • Variant: The changed version being tested against the control.
  • Random Assignment: Users are split into groups to reduce selection bias.
  • Primary Metric: The main success measure, such as conversion rate, sign-ups, revenue, or click-through rate.
  • Guardrail Metrics: Safety checks that ensure the change does not harm latency, errors, retention, support tickets, or user satisfaction.

  1. Define a clear hypothesis and business objective.
  2. Select the target audience, traffic split, and experiment duration.
  3. Choose primary, secondary, and guardrail metrics before launch.
  4. Run the test long enough to collect reliable data.
  5. Analyze results for practical impact, not only statistical significance.
  6. Ship the winning version, iterate, or stop if the result is inconclusive.

Users can be divided into experiment groups in several ways, depending on how stable, controlled, and targeted the test needs to be. The goal is to make sure each user consistently sees the same version while the experiment is running, and that the assignment can be measured later in analytics.

Common Ways to Divide Users
  • Pure Random Assignment: Each eligible user is randomly placed into control or variant groups at assignment time. This reduces selection bias, but the result must be stored in a cookie, database, or user profile so the user does not switch versions on every visit.
  • Deterministic Hash-Based Bucketing: A stable identifier, such as user ID or device ID, is passed through a hash function and mapped to a bucket. The same identifier always produces the same assignment, so the user consistently sees the same version.
  • Experiment-Specific Deterministic Bucketing: The experiment key is included in the hash input, such as checkout_button_copy_v2:user_123. This lets the same user be in different buckets for different experiments while staying stable inside each experiment.
  • Database-Persisted Assignment: The system stores the assigned variant in a database table or user profile after the first assignment. This gives strong consistency and auditability, especially when assignments must be shared across frontend, backend, email, and mobile experiences.
  • Hybrid Approach (Hashing + Persistence): The system first uses hashing to calculate a stable assignment, then persists that result. This combines predictable initial assignment with a durable record for analytics, debugging, customer support, and future analysis.
  • Percentage-Based Bucketing: Users are mapped into numeric buckets such as 0-99, then assigned by rollout percentage. For example, buckets 0-9 can receive a 10% variant rollout, while the remaining buckets stay in control.
  • Feature Flags: A feature-flag system decides which version a user sees. The flag can be configured by percentage, country, plan, user role, or allowlist, and can be turned off quickly without redeploying code.
  • Cookie or Session-Based Assignment: Anonymous users are grouped using browser cookies or session identifiers. This is useful before login, but users may switch groups if cookies are cleared, browsers change, or devices change.
  • Cohort-Based Assignment: Users are grouped by shared traits such as region, device type, plan, signup date, or customer segment. This is helpful when testing a feature for a specific audience, but the analysis should account for the fact that cohorts may behave differently.
  • Geo-Based Assignment: Different countries, states, cities, or data regions receive different versions. This is useful for market-specific rollouts, but geography can introduce external factors such as language, pricing, holidays, and traffic source differences.
  • Account or Organization-Level Assignment: All users in the same company, workspace, tenant, or household see the same version. This avoids inconsistent experiences inside shared accounts and is common in B2B SaaS products.
  • Invite or Allowlist Assignment: Only selected users, beta customers, employees, or internal testers receive the variant. This is useful for early validation before a wider A/B test, but it should not be treated as unbiased experiment data.
Implementation Approach
  1. Define the experiment: Give the experiment a stable key such as checkout_button_copy_v2, define the variants such as control and variant, and decide the traffic split.
  2. Choose the assignment unit: Decide whether assignment should happen by user ID, account ID, device ID, or anonymous cookie. Use account ID when multiple users share the same workspace or organization.
  3. Check eligibility: Apply targeting rules before assignment, such as country, plan, platform, app version, or login status. Users who are not eligible should stay outside the experiment.
  4. Assign a stable bucket: Convert the assignment unit and experiment key into a deterministic number, usually with a hash function. Map that number to a variant based on the traffic split.
  5. Persist the assignment: Store the chosen variant in a database, feature-flag platform, cookie, local storage, or analytics identity profile so the user keeps seeing the same version.
  6. Render the correct version: The application reads the assignment and shows the control or variant experience. Avoid changing the assignment midway through a user journey.
  7. Track exposure and outcomes: Log an exposure event when the user actually sees the experiment, then track conversion, revenue, retention, errors, latency, and other success or guardrail metrics.
  8. Analyze and decide: Compare results between groups after enough data is collected. Decide whether to ship, roll back, continue testing, or run a follow-up experiment.
Simple Bucketing Example

A common implementation is deterministic hashing. The same user and experiment key always produce the same bucket, so assignment stays stable across sessions. 

function hashToBucket(value) {
  let hash = 0;

  for (let i = 0; i < value.length; i++) {
    hash = (hash * 31 + value.charCodeAt(i)) % 100;
  }

  return hash;
}

function getVariant(userId, experimentKey) {
  const input = `${experimentKey}:${userId}`;
  const bucket = hashToBucket(input); // Stable number from 0 to 99

  if (bucket < 50) {
    return "control";
  }

  return "variant";
}

For a 10% rollout, buckets 0-9 can receive the variant and the rest can stay in control. For a 50/50 test, buckets 0-49 can be control and 50-99 can be variant.

Feature Flag Flow
  1. Create a flag for the experiment, such as new_checkout_enabled.
  2. Add targeting rules, such as only logged-in users on the latest app version.
  3. Configure the rollout percentage or variant split in the flag system.
  4. Read the flag value in the frontend or backend before rendering the feature.
  5. Send the assigned variant with analytics events so results can be grouped correctly.
  6. Increase rollout, pause the experiment, or disable the variant based on the results.
Important Implementation Notes
  • Use stable assignment: A user should not see control on one visit and variant on the next unless the experiment intentionally changes.
  • Do not assign after conversion: Assignment should happen before the user can complete the measured action, otherwise results can be biased.
  • Track exposure separately: Being eligible is not the same as seeing the experiment. Measure users who were actually exposed to the tested experience.
  • Avoid mixed experiences: Keep assignment consistent across frontend, backend, emails, notifications, and mobile apps when they are part of the same journey.
  • Handle anonymous-to-logged-in users: Decide how to merge a cookie-based assignment with a logged-in user ID so the user does not unexpectedly switch versions.
  • Prevent overlapping conflicts: Avoid running multiple experiments that change the same screen, price, recommendation, or checkout flow at the same time.
  • Keep an audit trail: Store the experiment key, variant, start date, end date, targeting rules, and rollout changes so the result can be explained later.

  • Test one meaningful change at a time when you need clear cause and effect.
  • Avoid stopping early just because the result looks positive after a small sample.
  • Check that user assignment is stable so the same user keeps seeing the same version.
  • Segment results by device, geography, traffic source, or user type when behavior may differ.
  • Watch for novelty effects, seasonality, tracking bugs, and overlapping experiments.
  • Document the hypothesis, result, decision, and follow-up learning for future teams.
