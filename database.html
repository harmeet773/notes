Database

Row level security controls which rows a user can read, insert, update, or delete from a database table. Instead of protecting only the whole table, access rules are applied row by row.

Why It Is Used
  • To ensure users can access only their own records.
  • To isolate tenant data in multi-tenant applications.
  • To enforce security rules directly inside the database.
  • To reduce the risk of accidental data exposure from application code.
Common Example

In a notes application, every note may have a user_id. A row level security policy can allow a logged-in user to view only rows where user_id matches their own user id.

Important Points
  • Enable row level security on tables that store user-specific or sensitive data.
  • Create separate policies for read, insert, update, and delete actions.
  • Test policies with different user roles before moving to production.
  • Do not rely only on frontend checks for data protection.
