mlflow.pyfunc is MLflow's generic Python model flavor. It lets you wrap custom inference logic in a standard predict() interface, so the model can be logged, loaded, evaluated, registered, and deployed consistently even when it is not tied to a specific framework like scikit-learn or TensorFlow.In Databricks (via MLflow), pyfunc.predict is the method you call on a loaded PyFunc model to perform inference.

In pyfunc models, the predict method is responsible for the entire flow, including pre-processing of inputs, calling the model for inference, and post-processing the outputs. This ensures all the logic for transformation and prediction is self-contained within the pyfunc model.

load_context is a required function in mlflow.pyfunc.PythonModel that loads artifacts (like model weights, tokenizers) once when the model is constructed, not per prediction request. This minimizes artifacts loaded during predict, speeding up inference by ensuring heavy loading happens only once at initialization.

mlflow.genai.evaluate() is MLflow's API for systematically evaluating GenAI applications such as RAG assistants and AI agents. It runs your model against test data, applies evaluation scorers for criteria such as factual accuracy, completeness, and usefulness, and logs results so quality can be tracked over time.

The function requires three main components: a dataset of test cases, scorers that define the evaluation criteria, and the model's predict() function that generates responses.

Its key importance is support for human Subject Matter Expert (SME) feedback alongside automated evaluation. Pure LLM-as-judge approaches can miss domain-specific nuances, so MLflow lets teams integrate calibrated expert judgments directly into the evaluation workflow. This hybrid approach combines the scale of automation with the domain accuracy of human experts, helping evaluations capture real customer-support quality signals that automated metrics alone may miss. Teams can also validate that automated LLM judges align with expert standards by training or calibrating them against SME annotations.

A Feature Store is a centralized repository that stores and manages reusable features (data attributes) used by AI, machine learning, and Generative AI applications. In Databricks, a Feature Store is typically built on top of Delta tables and provides additional capabilities such as feature discovery, governance, lineage tracking, and consistent access to features during both training and inference. For example, in a customer support chatbot, features like a user's subscription tier, preferred language, and account status can be stored in the Feature Store and retrieved to personalize prompts sent to an LLM. Similarly, in a recommendation system, features such as purchase history, average spending, and product preferences can be shared across multiple models instead of being recreated each time. In GenAI applications, the Feature Store often acts as a reliable source of structured context that complements unstructured data retrieved through RAG systems.

A Feature Store is a centralized place to create, discover, reuse, and govern machine learning features. Features are curated data columns, such as customer lifetime value, average transaction amount, or recent login count, that are used as inputs for model training and prediction.

In Databricks, feature tables are usually stored as Delta tables and governed through Unity Catalog. This helps data scientists and engineers share trusted features across teams while keeping permissions, lineage, and documentation consistent.

  • Training consistency: The same feature definitions can be used for model training and batch or real-time inference, reducing training-serving skew.
  • Reuse: Teams can search for existing features instead of rebuilding the same transformations in multiple projects.
  • Governance: Feature ownership, access control, lineage, and metadata can be managed with the rest of the lakehouse data.
  • Serving support: Features can be published for low-latency online lookup when models need fresh values during prediction.

Feature Serving allows you to expose ML features and real-time data to models via low-latency APIs.

Feature Serving is the capability that makes machine learning features available to applications and models in real time with low latency. While a Feature Store focuses on creating, organizing, and governing features, Feature Serving focuses on delivering the latest feature values during inference. This allows applications to retrieve up-to-date features without directly querying large analytical tables.

In Databricks, Feature Serving enables online access to features stored in feature tables managed through Unity Catalog. When a prediction request arrives, the serving layer can quickly fetch the required feature values and provide them to the model, ensuring that inference uses the same trusted feature definitions that were used during model training.

For example, an e-commerce recommendation system may need a customer's latest purchase count, average order value, and loyalty status before generating product recommendations. Instead of recalculating these values for every request, Feature Serving retrieves the latest feature values and supplies them to the model in milliseconds. Similarly, a fraud detection system can use recently updated transaction statistics and risk indicators to make real-time decisions.

In Generative AI applications, Feature Serving can provide structured context that complements information retrieved through RAG systems. For example, a customer support assistant may retrieve user-specific features such as subscription level, preferred language, account status, or recent activity and include them in prompts sent to an LLM, enabling more personalized and context-aware responses.

  • Low-latency access: Features can be retrieved quickly for online prediction workloads and real-time applications.
  • Training-serving consistency: The same feature definitions used during model training can be used during inference, reducing training-serving skew.
  • Real-time decision making: Applications can use the latest available feature values when generating predictions.
  • Scalability: Feature Serving is designed to handle high volumes of prediction requests while maintaining performance.
  • Simplified integration: Applications and model serving endpoints can retrieve features through managed APIs instead of directly accessing underlying data tables.

Feature Serving is commonly used alongside Model Serving. A typical workflow involves an application sending a request, Feature Serving retrieving the required feature values, and the model using those features to generate a prediction. This architecture helps ensure that production models have access to fresh, consistent, and governed data.

Although Feature Serving can deliver embedding vectors stored in feature tables, it is not designed for large-scale similarity search. For semantic search, retrieval-augmented generation (RAG), and nearest neighbor queries, Databricks Vector Search provides specialized vector indexing and retrieval capabilities. Feature Serving and Vector Search often work together, with Feature Serving supplying structured features while Vector Search retrieves relevant unstructured context.

Embeddings, Delta Tables, and Vector Search in Databricks

Where Are Embeddings Stored?

In Databricks, embeddings are typically stored directly inside a Delta Lake table.

The embedding is usually stored in an ARRAY<FLOAT> column together with:

  • Document text or chunk
  • Document ID
  • Metadata
  • Source information
  • Timestamps
  • Embedding vector

Embeddings live in the Delta Table.
The Vector Search Index is built from those embeddings.

How the Index Is Built

  1. Generate embeddings for document chunks.
  2. Store chunks and embeddings in a Delta table.
  3. Create a Vector Search Index on the Delta table.
  4. Databricks builds optimized vector-search structures.
  5. User query is converted into an embedding.
  6. The index finds the nearest matching vectors.
  7. Relevant rows are returned from the underlying Delta data.

Why Do We Need a Vector Search Index?

A Delta table may contain millions of embeddings. Calculating cosine similarity against every row would be slow and expensive.

The Vector Search Index uses Approximate Nearest Neighbor (ANN) algorithms to find similar embeddings efficiently, often in milliseconds.

Rule of Thumb

Delta Table = Storage Layer
Stores documents, metadata, and embeddings.

Vector Search Index = Retrieval Layer
Provides fast semantic search over the stored embeddings.

Interview Answer

Embeddings are typically stored in a Delta Lake table as an ARRAY<FLOAT> column. The Delta table acts as the source of truth and stores the document text, metadata, and embeddings. A Vector Search Index is then created on top of the Delta table to enable efficient similarity search. The index does not replace the Delta table; it is an optimized retrieval structure built from the embeddings stored in Delta Lake.

Aspect-Based Sentiment Analysis, or ABSA, is a more detailed form of sentiment analysis. Instead of only deciding whether the whole text is positive, negative, or neutral, ABSA identifies specific aspects mentioned in the text and finds the sentiment for each aspect.

For example, in the review "The model response was accurate, but the answer was too slow", normal sentiment analysis may mark the review as mixed. ABSA breaks it into aspects: accuracy is positive, while latency is negative.

In Gen AI applications, ABSA is useful for analyzing user feedback at a finer level. Teams can understand whether users are unhappy with answer correctness, response speed, tone, formatting, hallucinations, retrieval quality, or safety behavior. This makes it easier to improve the right part of the AI system instead of treating all negative feedback the same.

  • Aspect extraction: find the specific topic or feature mentioned in the feedback.
  • Sentiment classification: classify sentiment for each aspect as positive, negative, or neutral.
  • Root cause analysis: connect negative aspects to system components such as retrieval, generation, latency, grounding, or guardrails.

LLMs differ in size, architecture, modality, specialization, context window, latency, cost, and deployment model. Choosing the right model is usually a trade-off between answer quality, speed, governance, and operating cost.

General-purpose chat and reasoning models

These models are designed for broad language tasks such as question answering, summarization, writing, planning, coding help, and multi-step reasoning. They are a good default when the task is open-ended or when users may ask many kinds of questions.

Instruction-tuned models

Instruction-tuned models are trained to follow natural language directions. They are useful for assistants, enterprise workflows, extraction tasks, classification, and prompt-based automation where the model must reliably follow a requested format or policy.

Code models

Code-focused models are optimized for generating, explaining, translating, debugging, and reviewing code. They are commonly used in developer assistants, notebook copilots, SQL generation, test generation, and migration work.

Embedding models

Embedding models convert text, images, or other content into vectors that capture semantic meaning. They are usually used for search, recommendation, clustering, duplicate detection, and retrieval augmented generation rather than direct chat responses.

Multimodal models

Multimodal models can work with more than one input type, such as text, images, audio, video, or documents. They are useful for document understanding, chart interpretation, image analysis, call analysis, and workflows where important context is not only plain text.

Small language models

Small language models trade some capability for lower cost, lower latency, and easier deployment. They are useful for high-volume classification, routing, extraction, summarization, edge deployment, or tasks with narrow and well-defined behavior.

Open-weight and proprietary models

Open-weight models can often be hosted, fine-tuned, or governed inside an organization's own environment. Proprietary hosted models often provide strong out-of-the-box quality, managed scaling, and rapid feature updates. The best choice depends on data sensitivity, compliance, customization needs, cost, and operational complexity.

  • Use a stronger reasoning model for complex, ambiguous, or high-value tasks.
  • Use a smaller or specialized model for repetitive, narrow, high-volume tasks.
  • Use embedding models with a vector index when the system must search private or domain-specific knowledge.
  • Use multimodal models when the answer depends on images, PDFs, screenshots, charts, audio, or video.
  • Compare models using task-specific evaluation data, not only public benchmark scores.

E. BERT (Bidirectional Encoder Representations from Transformers)

BERT is a language model developed by Google in 2018 for natural language processing (NLP) tasks. It uses a **bidirectional encoder-only** Transformer architecture, meaning it reads text in both directions (left-to-right and right-to-left) to understand context from surrounding words. BERT is pre-trained using **masked language modeling**, where it predicts randomly masked words in a sentence.

**Key characteristics:**

  • Architecture: Encoder-only (no decoder) — designed for understanding/classification, not text generation
  • Best use cases: Text classification (sentiment analysis, spam detection), question answering (QA with fixed answers), search query understanding, natural language inference
  • Not suitable for: Conversational chatbots or open-ended text generation, as it cannot generate sequential responses like "Hi, how can I help you?"

Unlike GPT models (decoder-only), BERT excels at tasks where you need to **analyze or classify** text rather than **generate** new text. For chatbot applications requiring conversational generation, decoder-only models like GPT are the appropriate choice.

Prompt Coverage measures whether a prompt fully covers the important instructions, constraints, data requirements, and output expectations needed for the task. It helps check if the prompt gives the model enough guidance to answer correctly and consistently.

For example, if a support chatbot prompt says "answer customer questions" but does not mention tone, grounding, escalation rules, safety restrictions, or the expected response format, then prompt coverage is low. In Gen AI evaluation, prompt coverage is useful for finding gaps before production. Poor coverage can cause hallucinations, inconsistent answers, unsafe responses, or outputs that downstream systems cannot parse.

Beam Search Decoder is a decoding strategy used by language models to choose output text. Instead of always selecting only the single most likely next token, beam search keeps multiple possible sequences at each step and continues expanding the best candidates.

The number of candidate sequences kept is called the beam width. For example, if the beam width is 3, the decoder tracks the top three possible output paths, scores them, and finally returns the sequence with the best overall score.

Beam search can produce more coherent and optimized answers than greedy decoding because it considers future token combinations, not just the best immediate token. However, it can also reduce creativity and may produce repetitive or overly safe text if the search is too narrow or the scoring favors common phrases.

Beam search can be used in AI applications like QA bots, summarizers, and support assistants when the goal is to produce stable and high-confidence answers. It is useful when consistency matters more than creativity. However, for real-time QA bots, beam search may increase latency because the model has to evaluate multiple candidate sequences. Many modern chatbot applications instead use sampling settings like temperature and top-p, or lower-temperature decoding, depending on the required balance between speed, accuracy, and response variety.

  • Greedy decoding: selects the best next token at each step.
  • Beam search: keeps several likely sequences and chooses the best full sequence.
  • Higher beam width: explores more options but increases compute cost and latency.

Top P and Top K are decoding parameters used to control how a language model selects the next token. They are commonly used in AI applications like QA bots, chatbots, summarizers, and content generation tools to balance accuracy, creativity, and consistency.

Top K limits the model to choosing from only the K most likely next tokens. For example, if top_k is 50, the model only considers the top 50 candidate tokens and ignores the rest. A lower Top K makes responses more focused and predictable, while a higher Top K allows more variety.

Top P, also called nucleus sampling, selects from the smallest group of tokens whose combined probability reaches the value of P. For example, if top_p is 0.9, the model considers only the most likely tokens that together cover 90% of the probability mass. This makes Top P dynamic because the number of candidate tokens can change at each generation step.

In a QA bot, use lower randomness when the answer must be factual and grounded. A common setup is low temperature, top_p around 0.8 to 0.95, and a moderate top_k such as 40 or 50 if the serving API supports it. For creative writing or brainstorming, increase temperature and allow a higher top_p so the model can explore more varied answers.

Example generation settings:

{
  "temperature": 0.2,
  "top_p": 0.9,
  "top_k": 50,
  "max_tokens": 500
}
  • Use lower Top P or Top K for more deterministic and safer answers.
  • Use higher Top P or Top K when you want more diverse wording or creative output.
  • Do not increase all randomness settings together for factual QA, because it can increase hallucination risk.

Sequence Truncation with Token Replacement is a way to handle input text that is longer than the model's maximum context window. Instead of sending the full sequence, the system shortens it and replaces the removed part with a special marker token or summary token.

Normal truncation simply cuts off extra tokens. Token replacement is more informative because it leaves a signal that some content was removed. For example, a long document may be converted into an input like: 

Introduction text ... [TRUNCATED_SECTION] ... final question

This helps the model understand that the sequence is incomplete. In some systems, the replacement token can be a simple placeholder such as [TRUNCATED]. In more advanced systems, it can be a compressed summary, metadata marker, or special token learned during model training.

In Gen AI applications, this technique is useful when prompts include long conversation history, retrieved documents, logs, or user-provided files. It helps keep the input within the token budget while preserving the fact that omitted information existed.

  • Sequence truncation: removes tokens that exceed the model's context window or token budget.
  • Token replacement: inserts a marker or compressed representation where content was removed.
  • Main benefit: prevents context window overflow while reducing confusion caused by silent cuts.
  • Main risk: if important details are removed, the model may still miss key facts even though a marker is present.

The fundamental difference between single-turn and multiturn systems lies in memory management—specifically, whether the system is stateless or stateful. A single-turn system treats every interaction as an isolated event, while a multiturn system tracks ongoing context over a series of exchanges.

User: What about tomorrow?         -> AI: Error: Please specify a location.

Multiturn (Stateful):
User: What's the weather in Tokyo? -> AI: It is sunny and 72°F.
User: What about tomorrow?         -> AI: Tomorrow in Tokyo, expect rain and 68°F.

Multiturn systems rely on continuous context tracking to maintain a running history of the conversation slot-by-slot, making interactions feel natural and human-like rather than rigid or transactional.

  • Single-Turn (Stateless): Processes each query in total isolation, with zero memory of past prompts or responses.

  • Main risk: Requires active context management; if the history grows too long or messy, the system can lose track of the core intent.

The following core metrics and standard tools are used to evaluate Ranked Retrieval and Recommendation Systems. They measure how effectively an algorithm surfaces relevant items at the top of its returned lists.

1. MRR (Mean Reciprocal Rank)

MRR evaluates retrieval when the system only needs to find the first relevant item. It captures how high that first correct result appears by averaging its reciprocal rank position across all queries.

For example, if the first relevant document appears at rank 2 for one query and rank 1 for another, the reciprocal ranks are 1/2 and 1/1; the MRR is the average of those values. 

Input (queries and ranks of first relevant result):
Query A: first relevant at rank 3
Query B: first relevant at rank 1
Query C: no relevant result returned (treated as rank = ∞, reciprocal = 0)
Output: MRR = (1/3 + 1/1 + 0) / 3 = 0.444...
  • Focus: Order of the single first relevant item.
  • Interpretation: 0 to 1; higher means the very first result is more likely to be correct.
  • Best Use Case: "Factoid" search, question-answering systems, or "I'm feeling lucky" style queries where a user only needs one definitive answer.

2. MAP (Mean Average Precision)

MAP evaluates retrieval quality when queries have multiple relevant documents, and the system uses binary relevance (an item is either completely relevant or completely irrelevant). It penalizes systems that bury relevant items deep in the list. 

Input (precision at each relevant item's rank):
Query A: relevant items at rank 1 (Prec=1.0) and rank 3 (Prec=0.66) -> AP = (1.0 + 0.66) / 2 = 0.83
Query B: relevant item at rank 2 (Prec=0.5) -> AP = (0.5) / 1 = 0.5
Output: MAP = (0.83 + 0.5) / 2 = 0.665
  • Focus: Order and precision of all relevant items returned.
  • Interpretation: 0 to 1; higher scores mean the system successfully pushed more relevant items toward the top.
  • Best Use Case: Traditional web search engine evaluation or legal/medical document retrieval where finding all relevant pieces matters.

3. NDCG (Normalized Discounted Cumulative Gain)

NDCG evaluates retrieval quality using graded relevance scales (e.g., highly relevant, somewhat relevant, irrelevant). It sums up the "gain" of items, scales down (discounts) that gain for lower ranks, and normalizes it against a perfectly ordered list. 

Input (graded relevance scores on a 0-3 scale):
Query A actual output ranks: [Result 1 (score 3), Result 2 (score 0), Result 3 (score 2)]
Query A ideal output ranks:   [Result 1 (score 3), Result 3 (score 2), Result 2 (score 0)]
Output: NDCG = DCG of actual output / DCG of ideal output (ranges from 0.0 to 1.0)
  • Focus: Order based on multi-level degrees of relevance.
  • Interpretation: 0 to 1; 1 indicates the system perfectly sorted the items from most relevant to least relevant.
  • Best Use Case: Modern e-commerce product recommendations or streaming platforms (like Netflix/Spotify) where user interactions provide graded preferences (e.g., 1-5 stars, watch time percentages).

4. Evaluation Tool: trec_eval

trec_eval is the standard, globally recognized command-line tool used to evaluate Information Retrieval (IR) systems. Originally written in C for the Text REtrieval Conference (TREC), it computes all the core metrics mentioned above (MAP, MRR, NDCG, and more) by comparing a system's search results against a ground-truth "golden dataset."

To run an evaluation, the tool requires two specific plain-text input files: the qrels (ground truth) file and the results (system output) file. 

# Standard terminal execution syntax:
$ trec_eval [options] <qrels_file> <results_file>

# Example Output Snippet:
num_q          all    100
map            all    0.6650
recip_rank     all    0.4444
ndcg           all    0.7125
  • The Qrels File: Contains the true relevance judgments made by human assessors. Each line contains: [Query_ID] [Iteration] [Document_ID] [Relevance_Score].
  • The Results File: Contains the ranked output generated by your search engine algorithm. Each line contains: [Query_ID] [Q0] [Document_ID] [Rank] [Score] [System_Name].
  • Why it matters: It eliminates implementation discrepancies. Because different programming libraries may calculate metrics like NDCG with slight mathematical variations, trec_eval serves as the industry-wide baseline benchmark for publishing research papers and comparing search systems objectively.

Role-based masking in Unity Catalog is a data governance feature that allows organizations to hide, partially mask, or filter sensitive data based on the user's assigned role, group membership, or permissions. Instead of giving every user access to the original values, Unity Catalog can dynamically return masked or filtered values while still allowing authorized users to view the full data.

Can we perform both Row-level and Column-level masking?
Yes. Unity Catalog supports both Column-Level Masking (using Row Filters and Column Masks) to secure data at a granular level based on who is querying the data:

  • Column-Level Masking: Allows you to apply a masking function to a specific column. Security policies dictate whether a user sees the real data, partially masked data (e.g., credit card ends in 1111), or completely redacted data (e.g., NULL or fixed hashes), depending on their database roles or groups.
  • Row-Level Masking (Row Filtering): Allows you to filter out specific rows from a query result based on user privileges. For example, a regional sales manager might only see rows where the Region column equals 'US', while a global executive can see all rows.

For example, a customer table may contain email addresses, credit card information, and regional data. Data analysts may only see masked columns and filtered rows, while administrators or compliance teams can view the actual, complete dataset.

Autoscaling is a cloud computing feature that automatically adjusts computational resources (like CPUs, GPUs, or entire servers) based on real-time system demand without manual intervention. It continuously monitors metrics like CPU usage, memory, or request count, and dynamically adds or removes resources to match workload requirements.

For example, during a flash sale on an e-commerce site, traffic might spike 10x. Autoscaling automatically launches additional servers to handle the load, then terminates them when traffic returns to normal. 

Normal traffic: 2 servers running
Flash sale starts (10x traffic): System adds 8 more servers
Sale ends: System removes the 8 extra servers → back to 2 servers

Autoscaling is essential for running applications efficiently in the cloud. For LLM inference workloads (like RAG applications), it ensures sufficient GPU/compute capacity during high demand while avoiding over-provisioning during low-traffic periods.

  • Purpose: Automatically allocate/deallocate resources based on demand to maintain performance and optimize costs.
  • Two types: Horizontal scaling (adds/removes entire instances) and vertical scaling (increases/decreases CPU/RAM of existing instances).
  • Main benefit: Prevents over-provisioning (paying for unused capacity) while ensuring uptime during traffic spikes.
  • Main risk: Scaling lag (delay between detecting load and adding resources) can cause temporary slowdowns if thresholds are too aggressive.

When autoscaling increases resources for LLM inference, it adds more GPUs or servers to handle the workload. This improves response times and throughput during high demand, but it does NOT change the quality of answers or reduce the per-query cost of using the LLM.

For example, with 2 GPUs processing 100 requests/sec, latency might be 500ms with 10 requests queued. After scaling to 8 GPUs, latency drops to 150ms with 0 requests queued—but each query still costs the same per token. 

Before scaling: 2 GPUs, 100 req/sec → 500ms latency, 10 queued
After scaling:  8 GPUs, 100 req/sec → 150ms latency, 0 queued
Per-query cost: Still the same (same tokens, same price per token)

This distinction is critical for RAG applications. Autoscaling optimizes infrastructure utilization (not wasting idle capacity), but it doesn't reduce the actual LLM inference cost, which is determined by token count and number of queries.

  • What improves: Latency (faster responses), throughput (more concurrent queries), queue time (fewer waiting requests), and availability during traffic spikes.
  • What stays the same: Per-query cost, model quality, response accuracy, and token consumption—autoscaling doesn't make the LLM cheaper or smarter.
  • Main benefit for LLMs: Maintains performance under load without manual intervention, preventing dropped requests during peak usage.
  • Why it doesn't solve cost problems: LLM costs are driven by tokens × queries, not infrastructure. To reduce costs, you need prompt optimization (fewer tokens) or caching (fewer queries), not more GPUs.

The most effective way to control RAG application costs in Databricks without compromising response quality is to employ prompt optimization techniques and cache common query results. Prompt optimization reduces token usage per query, while caching eliminates redundant LLM calls for repeated questions.

For example, if users frequently ask "What is the return policy?", caching the response means the LLM only runs once, and subsequent identical queries return the cached answer instantly at near-zero cost. 

Without caching:
User asks "Return policy?" → LLM runs → 500 tokens → $0.01
User asks "Return policy?" → LLM runs → 500 tokens → $0.01
User asks "Return policy?" → LLM runs → 500 tokens → $0.01
Total: 1500 tokens, $0.03

With caching:
User asks "Return policy?" → LLM runs → 500 tokens → $0.01
User asks "Return policy?" → Cache hit → 0 tokens → $0
User asks "Return policy?" → Cache hit → 0 tokens → $0
Total: 500 tokens, $0.01 (67% savings)

Unlike autoscaling (which optimizes infrastructure but not per-query cost) or reducing max tokens (which risks incomplete answers), prompt optimization + caching directly reduces LLM usage while maintaining full response quality.

  • Prompt optimization: Use concise prompts, reduce top-k documents, summarize retrieved chunks, and encourage brief responses to cut token usage.
  • Caching: Store responses for common/similar queries in Databricks Cache or MLflow to avoid redundant LLM inference.
  • Main benefit: Directly reduces LLM inference costs by lowering token consumption and query count, without sacrificing answer quality.
  • Why other options fail: Autoscaling doesn't reduce per-query cost; checkpointing is for training (not inference); reducing max tokens truncates responses and harms quality.

Delta Lake is an open storage layer that adds reliability and database-like behavior to data lakes. It is commonly used in Databricks to build lakehouse tables on cloud storage such as AWS S3, Azure Data Lake Storage, or Google Cloud Storage.

A normal data lake may store files such as Parquet, CSV, or JSON, but file-based storage alone can be hard to manage when many jobs are reading and writing at the same time. Delta Lake solves this by adding a transaction log, called the Delta log, that tracks table versions and data changes. 

Data lake files + Delta transaction log = Delta Lake table storage

Because of this log, Delta Lake supports ACID transactions, schema enforcement, schema evolution, updates, deletes, merges, scalable metadata, and time travel. This makes it useful for reliable batch pipelines, streaming pipelines, analytics, machine learning, and BI workloads.

  • Purpose: make data lakes more reliable, consistent, and easier to query.
  • Core idea: store data files plus a transaction log that records every committed change.
  • Main benefit: multiple readers and writers can work with the same data safely.
  • Time travel: query older versions of data for debugging, audits, or rollback.
  • In Databricks: Delta Lake is the default foundation for many managed and external lakehouse tables.

A Delta table is a table stored in Delta Lake format. It keeps the actual data files, usually Parquet files, together with a transaction log that records every change made to the table.

In Databricks, Delta tables are used for reliable lakehouse data. The transaction log makes the table behave more like a database table than a plain folder of files. It supports ACID transactions, schema enforcement, updates, deletes, merges, and time travel. 

CREATE TABLE sales_delta (
  order_id BIGINT,
  customer_id BIGINT,
  amount DOUBLE
) USING DELTA;

A simple way to remember it: Delta Lake is the storage technology, and a Delta table is a specific table created using that technology. You can query it with SQL, process it with Spark, govern it with Unity Catalog, and use it in ETL, streaming, ML, and BI workloads.

  • Data files: store the table's rows, commonly in Parquet format.
  • Transaction log: tracks versions, commits, schema changes, and data modifications.
  • ACID support: helps reads and writes stay consistent even when multiple jobs run together.
  • Time travel: lets you query or restore older versions of the table.
  • Common use: bronze, silver, and gold tables in Databricks data pipelines.

The architecture of an AI summarization model is the set of components that convert a long document, conversation, or set of retrieved passages into a shorter version that preserves the most important meaning.

Modern summarization systems are usually built around a transformer-based language model. The input text is first cleaned, split into tokens, and passed through attention layers. These layers help the model understand which words, sentences, and sections are most relevant to the final summary.

A common abstractive summarization flow looks like this: 

Source text -> Preprocessing -> Tokenizer -> Transformer encoder/decoder or LLM -> Summary decoder -> Post-processing -> Final summary

In an encoder-decoder model, the encoder reads the source text and creates contextual representations. The decoder then generates the summary one token at a time. In a decoder-only LLM, the prompt contains the source text plus summarization instructions, and the model predicts the summary directly as the completion.

Production summarization systems often add extra layers around the model. For long documents, a chunking step breaks the input into smaller sections. Each section may be summarized first, and then a second model call combines those partial summaries into a final answer. This is often called map-reduce summarization.

  • Input layer: receives documents, transcripts, chats, emails, or retrieved passages.
  • Preprocessing layer: cleans text, removes noise, splits long inputs, and keeps useful metadata.
  • Tokenization layer: converts text into token IDs that the model can process.
  • Model layer: uses transformer attention to identify important context and generate or select summary content.
  • Decoding layer: controls the generated text using settings such as max tokens, temperature, Top P, and repetition penalties.
  • Post-processing layer: checks formatting, removes duplicates, enforces length, and may validate facts against the source.

There are three common summarization architecture patterns:

  • Extractive summarization: selects important sentences from the original text without rewriting them.
  • Abstractive summarization: generates a new, shorter version in natural language, often using different wording from the source.
  • RAG-assisted summarization: retrieves the most relevant chunks first, then summarizes only the grounded context.

The main design goal is to balance compression, factual accuracy, coverage, and readability. A good architecture should reduce the amount of text while keeping the key facts, decisions, dates, metrics, and relationships intact.

BLEU and ROUGE are automatic evaluation metrics used to compare generated text with one or more reference answers written by humans. They are useful when you need a quick, repeatable way to compare model outputs across a test dataset.

BLEU stands for Bilingual Evaluation Understudy. It is mostly used for machine translation. BLEU measures how many word sequences, called n-grams, in the generated text also appear in the reference text. It rewards precision, meaning the model gets a better score when the words it produces match the reference.

ROUGE stands for Recall-Oriented Understudy for Gisting Evaluation. It is mostly used for summarization. ROUGE measures how much of the reference summary is covered by the generated summary. It rewards recall, meaning the model gets a better score when it includes the important words, phrases, or sequences from the reference.

Mean Squared Error (MSE) is a different evaluation metric used for regression tasks, where the model predicts a numeric value instead of text. It computes the average of the squared differences between predicted and actual values, which means larger errors are penalized more heavily. MSE is commonly used for forecasting, price prediction, and other continuous-value outputs. 

Generated text + Reference text -> BLEU / ROUGE metric -> Similarity score
Metric Measures Best For Memory Hook
ROUGE Word Overlap Summarization ROUGE is for Recall
BLEU Translation Quality Machine Translation BLEU is for Bilingual
BERTScore Semantic Similarity Paraphrasing BERT is for Bonding
  • Use BLEU when evaluating translation or other tasks where exact wording and phrase matching are important.
  • Use ROUGE when evaluating summarization, especially when checking whether key points from a reference summary were included.
  • Use ROUGE-1 to compare individual word overlap between generated and reference summaries.
  • Use ROUGE-2 to compare two-word phrase overlap, which is stricter than ROUGE-1.
  • Use ROUGE-L to compare the longest matching word sequence, which helps measure sentence structure similarity.
  • Use MSE for regression or forecasting tasks where the output is a continuous number and you want to penalize larger errors more strongly.
  • Avoid using BLEU or ROUGE as the primary metric for open-ended chat, reasoning, code generation, creative writing, or factual QA where multiple very different answers can be correct.

These metrics are best for offline model evaluation, benchmark comparisons, regression testing, and choosing between prompt or model versions. For example, if you change a summarization prompt, ROUGE can help check whether the new prompt still preserves the same important content as the reference summaries.

BLEU and ROUGE should not be the only way to judge Gen AI quality. They rely heavily on word overlap, so they can penalize a correct answer that uses different wording and reward an answer that copies words but misses deeper meaning. For production systems, combine them with human review, factuality checks, task success metrics, and safety evaluations.

LLMs are deep learning models that need massive data and computing resources—far more than classical machine learning models. While classical models use clear metrics like accuracy or F1 score, evaluating LLMs is tougher because their output is language.

When evaluating LLMs, the first metric used is loss. During training, the model tries to guess missing words in sentences, and as it improves, the loss value decreases. This means the model becomes better at predicting the next word and understanding grammar. However, just like with phone autocomplete, the model may generate grammatically correct words that don’t always make sense. Loss measures word prediction accuracy but doesn’t guarantee truly coherent or meaningful sentences, so more advanced evaluation methods are needed to overcome these limitations.

To improve evaluation beyond just loss, another metric used is perplexity. Perplexity measures how confident a model is in picking a particular word from all possible choices— lower perplexity means the model is more certain. This helps produce more fluent sentences because the model is better at choosing words that fit together, so the text flows more naturally. However, even with low perplexity, the model might still generate sentences that sound good but aren’t actually relevant to the task or question. So, while perplexity is a step up from just measuring loss, it still doesn’t guarantee meaningful or accurate answers for specific tasks.

These metrics, while broadly useful, aren’t task specific—they don’t show how well a model does at things like question answering, summarization, or product recommendations. Metrics like loss, perplexity, and toxicity focus mainly on grammar and fluency, not on whether the sentences are actually useful for a particular task. This is a core challenge in Al system evaluation, since real-world tasks—like translation, code writing, or passing exams—require different benchmarks. Evaluation Metrics 1) context precision - Our first metric, Context Precision, is retrieval-related and measures the signal-to-noise ratio of the retrieved context. It is based on the relationship between the Query and the retrieved Context(s). Context Precision assesses whether the highly ranked chunks or nodes in the retrieval context are more relevant than irrelevant ones. For example, if your query is about Einstein's role in quantum mechanics, a high precision context would specifically mention his contributions to quantum theory, while a low precision context might broadly discuss his life and achievements without focusing on the query's specific technical area. 2) Context Relevancy - Context Relevancy is another retrieval-related metric that measures the relevancy of the retrieved context. Like precision, it is based on the Query and the Context(s). However, relevancy does not necessarily consider factual accuracy. Instead, it focuses on how well the retrieved context addresses the posed question. For example, if the query is about Einstein and quantum mechanics, context that details his initial challenges to quantum theory or his foundational ideas would be highly relevant, even if it doesn't represent the 'ground truth' of his final accepted contributions. Conversely, context about his pacifist views or citizenship status would be low relevancy. 3)Context Recall - Our third retrieval metric is Context Recall. Recall measures the extent to which all relevant entities and information are retrieved and mentioned in the context provided. This metric is based on comparing the Ground Truth against the retrieved Context(s). If the ground truth about Einstein mentions both his contributions to relativity and insights into quantum mechanics, a high-recall context will retrieve and mention both of those key facts. A low-recall context might only retrieve information about relativity, missing the quantum mechanics aspect entirely. 4)Faithfulness - Shifting now to generation-related metrics, we look at Faithfulness. This metric assesses the quality of the final generated response. Faithfulness measures the factual accuracy of the generated answer in relation to the provided context. It is specifically based on the relationship between the Response and the retrieved Context(s). A high faithfulness answer is supported by the context; for instance, if the context says Einstein was born on March 14th, a faithful answer will state March 14th. If the response provides an answer that contradicts the context—say, stating March 20th, it shows low faithfulness, regardless of whether March 20th might be factually correct outside of that specific context. 5) Answer Relevancy - it assesses how pertinent and applicable the generated response is to the user’s initial query. It is based on the alignment of the Response with the user's intent or the specific details of the Query. For example, if the query asks what Einstein is known for, a high relevancy answer will specifically mention his major theoretical achievements, such as developing the theory of relativity. A low relevancy answer might simply state he was a scientist, which is true but doesn't fully address the intent of the question. 6)Answer Correctness.- This metric measures the accuracy of the generated answer when compared directly to the ground truth. It is based on the Ground Truth and the Response. Answer Correctness encompasses both semantic and factual similarity with the ground truth. If the ground truth states Einstein won the Nobel Prize in Physics in 1921 for his work on the photoelectric effect, a high correctness answer will match those facts. Reasoning vs. Non-Reasoning Models

Effective prompting requires understanding the capabilities of the underlying model. Let's examine the two primary categories:

  • Non-Reasoning Models (e.g., Llama 3, GPT-4o): These models predict the next token based on statistical likelihood. They require explicit guidance, such as Chain of Thought (CoT) prompting ("Think step-by-step"), to break down complex logic and avoid rushing to incorrect conclusions.
  • Reasoning Models (e.g., OpenAI o1): These models are trained to generate their own internal chain of thought before producing a final answer. For these models, manual CoT prompting is often redundant or counterproductive. Context engineering for reasoning models focuses on defining the goal and constraints rather than the thinking process.

RAG vs. Retrieval Agent

RAG refers to the architectural pattern of coupling retrieval with generation, independent of any specific tooling or agent logic. Retrieval agents, by contrast, are concrete implementations that operationalize this pattern—handling query routing, retrieval orchestration, and context assembly within real systems.

To solve the knowledge boundaries we defined earlier, we shift from a memory-based approach to a context-based architecture known as Retrieval Augmented Generation (RAG). RAG injects proprietary or real-time data into the prompt, enabling the model to respond based on provided facts rather than its internal memory.

The RAG process consists of three key stages:

  • Retrieval: The system searches a knowledge base (indexed via Mosaic AI Vector Search) for relevant data chunks
  • Augmentation: The system injects these chunks into the context window
  • Generation: The model synthesizes an answer using only the injected data

Principles of Context Engineering -

Simply retrieving data is not enough; dumping raw information into a prompt often leads to confusion rather than clarity. As we move from basic RAG to production-grade systems, we must treat the context window not as a passive container, but as an engineered environment that actively shapes model behavior.

Context Engineering is the strategic design of the entire input window. It moves beyond writing a single instruction to managing the complete system state. We orchestrate the interplay between the System Instructions, Conversation History, Retrieved Data, and User Constraints to ensure the model has exactly the signal it needs to perform the task effectively.

Key components of an effective system prompt include:

  • Role Definition: Explicitly define the persona (e.g., "You are a Databricks Security Architect")
  • Negative Constraints: Define what the model cannot do (e.g., "Do not mention competitor products" or "Do not provide code unless explicitly requested")
  • Output Formatting: Enforce structured outputs (e.g., JSON, YAML, or Markdown tables) to ensure downstream applications can parse the response deterministically

Strict Grounding and Chunking

Retrieval must be strictly governed to prevent hallucinations and ensure accuracy. Key strategies include:

  • Grounding Instructions: Use explicit instructions to bind the model to the retrieved context. For example: "Answer using ONLY the provided context chunks. If the answer is not present, state 'I do not have that information.'"
  • Metadata Filtering: Utilize Unity Catalog metadata to filter retrieval before it reaches the model. For example, if a user asks about "2024 Revenue," the system should filter chunks where year=2024 to prevent the model from seeing outdated 2023 data

Managing Multi-Turn State

For applications involving long conversations (such as Agents), the context window will eventually fill up. Context Engineering requires strategic approaches to manage this state:

  • Summarization: Periodically compress the conversation history into a summary of key decisions and facts
  • Moving Window: Discard the oldest messages to free up token space for new retrieval
  • Selective Persistence: Determine which pieces of information (e.g., user name, current project ID) must remain in the context permanently versus what can be discarded

These strategies ensure that we maintain relevant context while staying within token limits.

Context Constraints: Token Budgets and Window Limits

Even the most elegantly engineered context is subject to the hard constraints of compute resources and model architecture. As we scale our applications, we must balance the desire for comprehensive context against the realities of token limits, latency, and operational costs.

Every model has a Context Window Limit (e.g., 8k, 32k, or 128k tokens). This represents the hard limit of working memory available to the model. The context window consists of:

  • Input Tokens: The text we send to the model (instructions + retrieved documents + history)
  • Output Tokens: The text the model generates

The Trade-off: As we fill the window with more retrieved data, the model's ability to reason degrades (the "Lost in the Middle" phenomenon), and latency increases significantly. Strategic context management is essential for maintaining performance.

Key considerations:

  • Cost Management: A naive RAG system that retrieves 50 documents for every query will burn through token budgets rapidly
  • Optimization Strategies:
    • Just-in-Time Retrieval: Instead of loading a full manual at the start of a chat, give the Agent a tool to retrieve specific sections only when the user asks a relevant question
    • Reranking: Use a reranker model to score the top 50 retrieved chunks and only inject the top 3-5 most relevant ones into the final context window

These strategies help us balance comprehensive context with cost efficiency and performance.

Key Takeaways:

  • Retrieval Agent Architecture: Use retrieval components to bridge knowledge gaps, and apply Context Engineering to optimize retrieval agents for performance, reliability, and cost.
  • Context is Finite: Manage the context window like a budget. Use filtering and reranking to maximize the value of every token.
  • Grounding is Mandatory: Strictly instruct the model to use only retrieved data and leverage Unity Catalog metadata to ensure that data is relevant and secure.

Search Methods

Different search methods serve different retrieval needs:

  • Similarity Search: This method retrieves content based on semantic correlation rather than exact word matching. It enables natural language queries to surface relevant results that may use different terminology.
    Text is converted into vector representations in the same embedding space, and then operations are peformed on the input vectore and the stored vectors to find the closest matches. Examples:
    • Query: "how to deal with anxiety" → Returns: "coping with PTSD," "managing stress," "panic attack relief"
    • Query: "best laptop for coding" → Returns: "computers for software development," "machines for programming," "developer workstations"
    • Query: "quick dinner recipes" → Returns: "fast meal ideas," "30-minute dishes," "easy cooking recipes"
  • Full-Text Search: This traditional approach relies on keyword matching. It excels at finding specific terms like part numbers, product codes, or proper nouns, but fails to capture semantic intent or recognize synonyms.

    How It Performs Matching: Exact keyword/token matching — requires 100% word match. "app" won't find "apple" (no partial matching).
    Index Usage: Uses inverted text indexes for fast search (O(log n) performance).
    Stop Words: Filters out common words like "the", "and", "is".
    Stemming: Yes — reduces "running", "ran" to root form "run".

    Full-Text Search vs. Regex Matching Comparison:
    Aspect Full-Text Search Regex Matching
    Partial Match ❌ No — requires exact word ✅ Yes — "app" finds "apple"
    Index Usage ✅ Uses text indexes (fast) ❌ No index (scans all data)
    Performance 🔥 Much faster on large datasets 🐌 Slow on large datasets
    Stemming ✅ Yes ("running" → "run") ❌ No (must specify patterns)
    Flexibility Word-based, boolean operators Ultra-flexible: wildcards, anchors, groups
    Best For Product search, content search, large datasets Pattern validation, partial matches, small datasets

    When to Use Each:
    • Use Full-Text: E-commerce search ("wireless headphones"), blog/content search, large datasets, RAG applications
    • Use Regex: Email/phone validation, partial substring search ("app" in "apple"), SKUs with patterns, small datasets, no index setup needed

    Examples:
    • Query: "iPhone 15 Pro" → Returns only pages containing exact phrase "iPhone 15 Pro"
    • Query: "SKU-12345" → Returns only products with that exact part number
    • Query: "Python programming" → Returns pages with both "Python" AND "programming" keywords (not "coding" or "software development")
    • Query: "ESA-2024-B" → Returns only documents with that exact product code
    • Regex alternative: Query: `{ $regex: "app" }` → Finds "apple", "application", "applet" (partial match)
  • Hybrid Search: This powerful approach combines vector similarity search with keyword-based search. By leveraging both semantic understanding and exact keyword matching, hybrid search typically delivers higher retrieval accuracy than either method alone.

    How it works:
    • Vector search captures the semantic meaning of a query, so it can find related content even when the exact words do not match.
    • Keyword search focuses on exact terms such as product names, version numbers, locations, IDs, or domain-specific phrases.
    • Hybrid search combines both result sets and ranks them together, giving you the benefit of semantic relevance plus exact term precision.
    Why it is useful:
    • Improves retrieval quality when user queries contain both intent and exact identifiers.
    • Handles natural language questions better than pure keyword search.
    • Preserves important exact matches such as names, locations, codes, and model versions.
    • Works especially well in RAG systems where relevant context must be retrieved before sending it to the LLM.

    Examples:
    • Query: "MacDonald restaurant near Mumbai" → Returns: semantic match for "restaurant" + exact match for "Mumbai" location
    • Query: "Llama 3 model documentation" → Returns: semantic understanding of "AI model docs" + exact match for version "Llama 3"
    • Query: "best wireless headphones Sony WH-1000XM5" → Returns: semantic match for "headphones recommendations" + exact match for model "WH-1000XM5"
    • Query: "Databricks MLflow tutorial Python" → Returns: semantic understanding of "machine learning pipeline guide" + exact matches for "Databricks", "MLflow", and "Python"
    How to implement hybrid search:
    • Step 1: Store document text and metadata in a search system or vector database. Metadata can include title, category, author, tags, product name, version, location, etc.
    • Step 2: Generate embeddings for each document or chunk using an embedding model and store them in a vector index.
    • Step 3: Build a keyword index on the same documents using a keyword-based method such as BM25 or full-text search.
    • Step 4: At query time, run both searches in parallel:
      • perform vector similarity search using the query embedding
      • perform keyword search using the raw query text
    • Step 5: Merge and rank results by combining scores from both searches. This can be done using weighted scoring, reciprocal rank fusion (RRF), or custom ranking logic.
    • Step 6: Return the top ranked chunks/documents to the application or pass them to the LLM in a RAG workflow.
    Common scoring strategies:
    • Weighted score fusion: Final score = (vector score × weight) + (keyword score × weight)
    • Reciprocal Rank Fusion (RRF): Combines ranked lists from both methods without requiring score normalization
    • Metadata boosting: Increase rank if the result matches important metadata such as category, product name, date, or location
    Typical use cases:
    • Enterprise document search
    • RAG pipelines for chatbots and assistants
    • E-commerce product search
    • Support ticket and knowledge base retrieval

Metrics

To determine how "similar" two vectors are, we use two main types of metrics—distance metrics and similarity metrics—each suited to different retrieval scenarios.

Distance metrics are used when you want to quantify how far apart two vectors are in space—ideal for clustering, outlier detection, or when magnitude matters. Similarity metrics are used when you want to know how closely aligned two vectors are in direction—ideal for semantic search, document retrieval, and most NLP applications where meaning is more important than scale.

Distance Metrics

  • Euclidean Distance (L2): Measures the straight-line distance between two points in vector space. A lower value means vectors are more similar. Use this when you care about the absolute difference in all dimensions, such as clustering or anomaly detection.
  • Manhattan Distance (L1): Sums the absolute differences across all dimensions. A lower value means vectors are closer. This is useful when differences along each axis are equally important, such as in grid-based or sparse data.

Similarity Metrics

  • Cosine Similarity: Measures the cosine of the angle between two vectors. A higher score means greater similarity. This is the most popular metric for text embeddings because it focuses on orientation (semantic meaning) rather than magnitude, making it robust to document length or scale differences.

Search Strategies

Two primary strategies balance accuracy and performance:

  • K-Nearest Neighbors (KNN): An exact search method that calculates the distance between the query vector and every vector in the database. While highly accurate, it's computationally expensive and doesn't scale well to large datasets—imagine comparing your query against millions of documents one by one.
  • Approximate Nearest Neighbors (ANN): A strategy that trades a small amount of accuracy for dramatic speed gains. ANN uses sophisticated indexing algorithms like HNSW (Hierarchical Navigable Small Worlds) or FAISS (Facebook AI Similarity Search) to navigate vector space efficiently, checking only a subset of vectors while still finding highly relevant results.

Reranking Pipeline

To bridge the precision gap in initial retrieval, we add a reranker to the pipeline:

  • Initial Retrieval: The vector store retrieves a broad set of candidate documents (typically the top 20 to 50) using fast ANN algorithms.
  • Reranking: A specialized model (often a Cross-Encoder) evaluates the actual relevance of each candidate document against the specific query, considering their relationship in detail.
  • Reordering: Documents are re-sorted based on the reranker's relevance scores, placing the most pertinent information at the top for the language model to process.
